

# Защита данных: только администратор может изменять контент

## Проблема

RLS-политики на таблицах `blog_posts`, `courses`, `faq`, `testimonials` разрешают INSERT/UPDATE/DELETE **любому** аутентифицированному пользователю. Даже без формы регистрации на сайте, злоумышленник может создать аккаунт напрямую через API и получить полный доступ к изменению данных.

## Решение

Создать систему ролей и ограничить запись только пользователю с ролью `admin`.

## Что будет сделано

### 1. Миграция базы данных (одна SQL-миграция)

- Создать тип `app_role` (enum: `admin`, `user`)
- Создать таблицу `user_roles` с RLS
- Создать функцию `has_role()` (SECURITY DEFINER) для безопасной проверки ролей
- Назначить роль `admin` существующему пользователю (`8a78f168-b2f6-4dcf-8ba6-0a81c3c84065`)
- Заменить RLS-политики на всех 4 таблицах: INSERT/UPDATE/DELETE только для `has_role(auth.uid(), 'admin')`
- SELECT остается открытым (публичные данные)

### 2. Проверка роли на фронтенде

В `src/pages/Admin.tsx` после входа -- запрос к `user_roles`. Если роли `admin` нет, показать "Доступ запрещен" вместо панели управления. Это дополнение к серверной защите (RLS), а не замена.
